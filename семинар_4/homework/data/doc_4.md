# API и пользовательский интерфейс

API в проекте `rental_price_mlops` нужен для того, чтобы обученная модель могла использоваться как сервис. Это важный переход от экспериментального ML к MLOps: модель не просто лежит в файле и не запускается только из ноутбука, а принимает входные данные через веб-интерфейс или HTTP-запрос и возвращает прогноз цены аренды. В учебном проекте для этого используется FastAPI.

Сервис запускается командой `uvicorn rental_price_mlops.api.main:app`. После запуска пользователь может открыть несколько страниц. Главная страница доступна по пути `/`. Служебная проверка состояния находится на `/health`. Если `/health` возвращает корректный ответ, значит приложение запущено и готово к работе. Документация API доступна через `/docs`, где можно посмотреть endpoints и попробовать отправить запросы в интерактивном режиме.

Для пользователя особенно важна страница `/predictions-ui`. Она позволяет отправить параметры объекта аренды и получить предсказанную цену. Такой интерфейс делает проект понятным для демонстрации: преподавателю или пользователю не нужно писать Python-код, чтобы проверить модель. Достаточно открыть страницу, заполнить поля и посмотреть результат. Это превращает модель в прикладной инструмент.

Кроме страницы предсказаний в проекте есть `/monitoring-ui`. Эта страница нужна для вывода состояния мониторинга. Там можно показать, есть ли признаки дрейфа данных, какие показатели отклоняются от reference dataset и какие флаги аномалий сработали. В MLOps-проекте важно не только получить прогноз, но и понимать, насколько текущие входные данные похожи на данные обучения. Поэтому monitoring UI является одной из ключевых частей демонстрации.

Также в проекте есть `/experiments-ui`. Эта страница связана с результатами экспериментов, метриками и сравнением запусков. Она помогает показать, что модель может развиваться: можно обучать новые версии, сравнивать их качество и выбирать лучшую. Даже если в учебной версии эксперименты реализованы упрощенно, наличие отдельного интерфейса показывает правильную архитектурную логику.

API связан с подготовкой данных и обучением модели. Важно, чтобы данные, приходящие в API, проходили ту же логику обработки признаков, что и данные на этапе обучения. Если во время обучения использовались одни признаки, а в API передаются другие или иначе закодированные значения, предсказания будут некорректными. Поэтому в MLOps-проекте требуется единая схема признаков и воспроизводимый preprocessing.

API также связан с мониторингом. Каждый запрос может быть записан в лог или сохранен для дальнейшего анализа. На основе входных данных можно проверять data drift. На основе предсказаний можно смотреть, не изменилось ли распределение прогнозируемых цен. Если со временем появляются фактические значения, можно оценивать реальное качество модели и фиксировать concept drift. Таким образом, API становится источником данных для дальнейшего мониторинга.

При запуске сервиса в проекте возникала ошибка `Address already in use 8000`. Она означает, что порт 8000 уже занят другим процессом. Такая проблема часто возникает, если предыдущий uvicorn-процесс не был корректно остановлен. Решение состояло в том, чтобы найти процесс и завершить его через `kill -9`, после чего сервис можно было запустить заново. Этот пример показывает, что MLOps включает не только код модели, но и обслуживание приложения.

Для демонстрации проекта API лучше показывать после раздела с метриками. Сначала объясняется, что модель обучена и имеет измеримое качество. Затем показывается, что модель можно использовать через интерфейс. После этого логично показать мониторинг, потому что любой работающий сервис должен контролироваться. Такая последовательность делает защиту понятной: данные, модель, сервис, мониторинг, переобучение.

Пользовательский интерфейс не обязательно должен быть сложным. Для учебного проекта достаточно нескольких страниц, которые показывают главные функции системы. Главное, чтобы они были связаны с реальными компонентами: моделью, предсказаниями, мониторингом и экспериментами. Поэтому API и UI в проекте выполняют роль связующего слоя между технической моделью и прикладным использованием.
