# Ошибки запуска и их исправление

Во время работы над проектом `rental_price_mlops` возникали типичные технические ошибки, связанные с запуском сервисов, Docker, портами и состоянием репозитория. Такие ошибки важны для описания, потому что MLOps-проект включает не только модель, но и инфраструктуру. Умение диагностировать проблемы является частью работы с системой.

Одна из ошибок была связана с портом 8000. При запуске API через `uvicorn rental_price_mlops.api.main:app` появлялось сообщение `Address already in use`. Это означает, что порт уже занят другим процессом. Чаще всего причина в том, что предыдущий сервер uvicorn был запущен и не остановлен корректно. Для исправления нужно найти процесс, который занимает порт, и завершить его. В проекте проблема была решена через `kill -9`, после чего API удалось запустить снова.

Еще одна группа проблем была связана с Docker. Если Docker daemon не работает, команды Docker Compose не смогут запустить контейнеры. На Mac это может означать, что Docker Desktop закрыт или завис. В такой ситуации нужно сначала проверить, что Docker запущен, затем повторить команду. Это кажется простой проблемой, но она часто мешает демонстрации, потому что без Docker не открываются сервисы Prometheus и Grafana.

Команда `docker compose ps` использовалась для проверки состояния контейнеров. В одном из моментов она вывела пустую таблицу: `NAME IMAGE COMMAND SERVICE CREATED STATUS PORTS`, но без запущенных сервисов. Это означало, что контейнеры не работают или не были созданы. Такая проверка помогает понять, действительно ли инфраструктура поднята. Если список пустой, нужно запускать compose заново или смотреть логи.

Также возникала проблема с удалением или остановкой minikube. При выполнении `minikube delete` появлялась ошибка, связанная с Docker metadata и контейнером `minikube`. Сообщение указывало, что контейнер не был полностью удален и может потребоваться ручное удаление VM из hypervisor. Эта ошибка не является центральной для проекта аренды, но показывает, что локальное окружение может влиять на работу Docker и Kubernetes-компонентов.

В проекте также упоминалась страница `/retrain-ui`, которая в один момент не открывалась. Это значит, что компонент переобучения требовал проверки маршрутов, регистрации endpoint или запуска соответствующего сервиса. В таких случаях нужно сначала открыть `/health`, затем `/docs`, чтобы увидеть доступные endpoints, после чего проверить код API и логи сервера. Если другие страницы работают, проблема может быть точечной.

Git также использовался для диагностики состояния проекта. Команда `git status` показывала, что ветка `main` синхронизирована с `origin/main`, но есть измененный `dvc.lock` и неотслеживаемые папки `backup_before_dvc_pull/` и `k8s/`. Это важно, потому что перед завершением работы нужно решить, какие изменения коммитить. Не все локальные файлы должны попадать в репозиторий.

Иногда можно завершить работу на промежуточном этапе, если сервисы остановлены или состояние проекта понятно. Но перед этим нужно проверить, что важные изменения не потеряются. Полезно выполнить `git status`, сохранить нужные файлы, остановить Docker Compose, если он был запущен, и записать, с какого шага продолжать. Такой подход снижает риск того, что на следующий день будет непонятно, что уже было сделано.

Ошибки запуска также полезны для отчета и защиты. Они показывают, что проект был действительно запущен, а не только описан теоретически. Преподаватель может увидеть, что были реальные проблемы с портами, контейнерами и UI, а студент смог их диагностировать. Это повышает практическую ценность проекта.

Для RAG этот документ должен находиться по вопросам о технических сбоях, `Address already in use`, `kill -9`, Docker daemon, пустом `docker compose ps`, проблеме `/retrain-ui`, `git status` и minikube. Он хорошо подходит для вопросов типа «какие проблемы возникали и как они были решены».
