# Общее описание MLOps-проекта прогнозирования цены аренды

Этот документ описывает общий смысл учебного MLOps-проекта `rental_price_mlops`. Проект посвящен прогнозированию цены аренды жилья на основе табличного датасета `AB_NYC_2019.csv`. В качестве исходной бизнес-задачи рассматривается ситуация, когда сервису размещения объявлений или аналитической команде нужно быстро оценивать ожидаемую стоимость объекта аренды. Такая оценка может использоваться для проверки адекватности цены, анализа рынка, построения рекомендаций для владельцев жилья и демонстрации полного жизненного цикла ML-системы.

Главное отличие проекта от обычного ноутбука с моделью состоит в том, что здесь моделирование оформлено как воспроизводимая система. Проект включает подготовку данных, разделение выборки, обучение модели, сохранение артефактов, API для инференса, пользовательские страницы, мониторинг, Docker-инфраструктуру, DVC и Git. Поэтому акцент сделан не только на качестве предсказаний, но и на том, как модель может быть запущена, проверена, обновлена и показана пользователю.

Проект построен вокруг нескольких логических модулей. Первый модуль отвечает за данные. Он читает исходный файл `AB_NYC_2019.csv`, обрабатывает признаки, делит данные на train, validation и test, а затем сохраняет подготовленные выборки в формате parquet. Отдельно создается reference dataset, который нужен для сравнения будущих входных данных с базовым распределением. Это важно для мониторинга дрейфа данных.

Второй модуль отвечает за обучение модели. Он использует подготовленные parquet-файлы, обучает регрессионную модель и сохраняет итоговые метрики. В проекте отдельно рассматриваются метрики в логарифмической шкале и в исходной шкале цены. Последние зафиксированные результаты показывают, что модель неплохо улавливает часть закономерностей в логарифмированной цене, но хуже объясняет разброс в исходных денежных значениях. Это типично для задачи с арендой, где есть выбросы, дорогие объекты и сильная неоднородность районов.

Третий модуль связан с API и интерфейсом. Модель можно запускать не только из Python-скрипта, но и через веб-сервис. Для этого используется FastAPI и команда запуска через `uvicorn`. В проекте доступны служебные и пользовательские страницы: `/health`, `/docs`, `/predictions-ui`, `/monitoring-ui`, `/experiments-ui`. Эти страницы позволяют показать, что система не ограничивается обучением модели, а может принимать запросы, возвращать прогнозы и выводить результаты мониторинга.

Четвертый модуль связан с MLOps-инфраструктурой. В проекте используется Docker Compose, который помогает запускать сервисы в одном окружении. Для мониторинга используются Prometheus и Grafana. Prometheus собирает метрики, а Grafana визуализирует их в виде дашбордов. Такая схема делает проект ближе к реальной промышленной ML-системе, где важно видеть состояние сервиса, а не только итоговую точность модели.

Пятый модуль связан с хранением артефактов и воспроизводимостью. Для этого используются Git и DVC. Git отвечает за код, конфигурации и небольшие текстовые файлы. DVC помогает управлять данными, моделями и артефактами, которые не стоит хранить напрямую в репозитории. В проекте также упоминалось хранение `mlflow_artifacts` и `dvc_storage` на внешнем диске `ARCHIVE`, чтобы не перегружать основной диск и репозиторий.

На защите такой проект можно показывать как полный пример MLOps-системы. Сначала демонстрируется подготовка данных, затем обучение модели и метрики, после этого API и пользовательский интерфейс. Далее можно показать мониторинг, флаги аномалий, таблицу последних предсказаний, эксперименты и сценарий переобучения. Именно эта связка делает проект цельным: пользователь видит не отдельный алгоритм, а систему, которая может работать после запуска.

Корпус для RAG в рамках домашнего задания построен именно на материалах этого проекта. Это удобно, потому что тексты содержат разные, но связанные темы: данные, модель, API, Docker, мониторинг, переобучение, DVC и ошибки запуска. Благодаря этому можно сформулировать как простые вопросы, так и сложные multi-hop вопросы, где ответ требует найти информацию сразу в двух или трех документах.
