# DVC, Git и хранение артефактов

В проекте `rental_price_mlops` Git и DVC выполняют разные, но связанные роли. Git используется для хранения кода, конфигураций, небольших текстовых файлов и истории изменений. DVC используется для данных и артефактов, которые могут быть слишком большими или слишком часто изменяться для обычного Git-репозитория. Такое разделение является стандартным для MLOps-проектов.

Git удобен для отслеживания изменений в Python-файлах, README, конфигурациях, docker-compose-файлах и скриптах. Команда `git status` позволяет увидеть, какие файлы изменены, какие еще не добавлены в индекс и что будет закоммичено. В проекте уже возникала ситуация, когда `git status` показывал измененный `dvc.lock` и новые папки `backup_before_dvc_pull/` и `k8s/`. Это нормальный рабочий процесс: перед коммитом нужно понять, какие изменения относятся к задаче, а какие являются временными или служебными.

DVC нужен для воспроизводимости данных и моделей. Если исходный датасет, обработанные parquet-файлы, обученная модель и отчеты хранятся хаотично, сложно понять, какие результаты относятся к какой версии данных. DVC позволяет связать данные, этапы обработки и артефакты в единый pipeline. Файл `dvc.lock` фиксирует конкретное состояние зависимостей и результатов. Если он изменился, это обычно означает, что изменилось состояние DVC-пайплайна.

В проекте использовались внешние хранилища для артефактов. Упоминались `mlflow_artifacts` и `dvc_storage` на внешнем диске `ARCHIVE`. Такое решение связано с ограничениями локального диска и размером файлов. Данные, модели и логи могут быстро занимать много места. Если хранить все в Git, репозиторий станет тяжелым и неудобным. Поэтому тяжелые объекты лучше выносить во внешнее DVC-хранилище.

Важно понимать, что DVC не заменяет Git. DVC работает вместе с Git. Обычно в Git попадают `.dvc`-файлы, `dvc.yaml`, `dvc.lock` и код, а сами данные лежат в DVC remote. Это позволяет другому пользователю клонировать репозиторий, установить зависимости и подтянуть данные через DVC. В результате проект становится воспроизводимым без хранения тяжелых файлов внутри GitHub.

DVC также полезен для переобучения. Когда модель переобучается на новых данных, нужно сохранить новую версию артефактов. Если метрики изменились, важно понимать, из-за чего это произошло: из-за нового кода, новых данных или новых параметров. Git помогает увидеть изменения в коде, DVC помогает увидеть изменения в данных и результатах. Вместе они дают контроль над жизненным циклом модели.

В учебном проекте DVC также связан с безопасностью репозитория. Не стоит загружать большие временные папки, локальные кэши и приватные данные напрямую в GitHub. Перед коммитом нужно проверять `git status` и осознанно добавлять только нужные файлы. Например, папка `backup_before_dvc_pull/` может быть полезной локально, но не всегда должна попадать в публичный репозиторий.

GitHub в этом проекте используется как место для сдачи и демонстрации кода. Основная ветка называется `main`. У пользователя есть собственный форк, а репозиторий преподавателя подключен как `upstream`. Такая схема позволяет подтягивать новые материалы курса из `paNikitin/gen-ai` и пушить свои решения в `KShuvalova/gen-ai`. Это важно для учебных домашних заданий, но принцип похож на работу с upstream в реальных open-source проектах.

Хранение артефактов также связано с Docker. Если контейнеры создают логи, модели или отчеты, нужно заранее решить, где эти файлы будут лежать. Без volumes артефакты могут исчезнуть после удаления контейнера. Поэтому инфраструктура Docker и DVC должны быть согласованы. Docker отвечает за запуск окружения, DVC отвечает за данные и артефакты, Git отвечает за код.

Для защиты проекта можно объяснить, что Git, DVC и внешнее хранилище делают проект воспроизводимым. Это значит, что результаты можно повторить, а не просто показать один раз на локальной машине. Если преподаватель спросит, почему модель и данные не лежат целиком в GitHub, ответ состоит в том, что Git не предназначен для тяжелых артефактов, а DVC решает эту задачу лучше.

Для RAG этот документ является основным источником по вопросам о DVC, Git, `dvc.lock`, внешнем диске `ARCHIVE`, `mlflow_artifacts`, `dvc_storage`, upstream и origin. В multi-hop вопросах он может извлекаться вместе с документом про Docker или переобучение.
