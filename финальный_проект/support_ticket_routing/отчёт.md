# Отчёт по финальному проекту

## 1. Задача и данные

Тема проекта: AI-конвейер для классификации и маршрутизации тикетов поддержки.

Цель проекта состоит в разработке воспроизводимой системы, которая получает текст обращения клиента, определяет тип проблемы, приоритет, ответственный отдел, необходимость эскалации и подбирает релевантные правила обработки обращения. Дополнительно система проверяет, что evidence quote действительно присутствует в исходном тикете, а использованные источники существуют в базе policy-документов.

В качестве источника данных использован датасет Customer Support Tickets Dataset 200k Records с Kaggle. Исходный датасет содержит 200000 строк. Для финального проекта была создана уменьшенная обезличенная подвыборка из 100 тикетов. Персональные поля `customer_name` и `customer_email` были исключены и не попадают в рабочий файл проекта.

В проекте используются следующие входные данные:

- `input/tickets.csv` - 100 обезличенных тикетов;
- `input/eval_cases.json` - 20 размеченных eval-кейсов;
- `input/challenge_cases.json` - 12 сложных кейсов для проверки устойчивости;
- `input/policy_docs/` - 7 markdown-документов с правилами маршрутизации.

Технически система поддерживает 8 категорий тикетов:

- `account_access`;
- `billing_refund`;
- `technical_bug`;
- `subscription`;
- `security_privacy`;
- `delivery_status`;
- `product_question`;
- `other`.

В рабочей подвыборке из Kaggle представлены 4 основные категории:

- `account_access`;
- `billing_refund`;
- `subscription`;
- `technical_bug`.

Поддерживаемые ответственные отделы:

- `support_l1`;
- `support_l2`;
- `billing_team`;
- `technical_team`;
- `security_team`;
- `logistics_team`;
- `retention_team`.

## 2. Архитектура системы

Pipeline состоит из нескольких последовательных этапов.

Первый этап - подготовка данных. Скрипт `prepare_data.py` читает raw CSV, удаляет персональные поля, нормализует категории и приоритеты, формирует `tickets.csv`, `eval_cases.json` и `dataset_profile.json`.

Второй этап - structured output. В файле `schema.py` определены Pydantic-схемы для входного тикета, результата классификации, RAG-документов, agent trace, hallucination check, judge result и eval result. В проекте используются `field_validator`, которые проверяют непустой текст обращения, корректность source_id, длину evidence quote, наличие used_sources и наличие шагов agent trace.

Третий этап - RAG. В файле `rag.py` реализован поиск по policy-документам. В качестве retrieval-механизма используется TF-IDF через `TfidfVectorizer` и cosine similarity. Для каждого тикета система возвращает top-3 релевантных policy-документа.

Четвёртый этап - agent tools. В файле `tools.py` реализованы инструменты:

- `classify_ticket_text`;
- `choose_department`;
- `should_escalate`;
- `retrieve_policy_docs`;
- `hallucination_check`.

Пятый этап - агент. В файле `agent.py` система последовательно вызывает инструменты, формирует результат маршрутизации и сохраняет trace. Для каждого тикета фиксируются пять шагов: классификация, выбор отдела, проверка эскалации, retrieval policy-документов и проверка галлюцинаций.

Шестой этап - оценка. В `eval.py` система сравнивает предсказания с 20 размеченными eval-кейсами. Оцениваются category accuracy, priority accuracy, department accuracy, escalation accuracy, quote valid rate, RAG hit rate, корректность agent path, ghost quote count и fake source count.

Седьмой этап - LLM-as-judge. Основной классификатор в проекте является rule-based, а LLM используется для независимой judge-оценки результатов. В `llm_judge.py` реализована проверка результатов через DeepSeek-compatible API. Судья получает текст тикета, ожидаемые значения, предсказание системы и найденные policy-документы. Он возвращает JSON-оценку по категориям: category_ok, priority_ok, department_ok, evidence_ok, source_ok и judge_pass. Ответ LLM валидируется через Pydantic-модель `LLMJudgeResponse`; при ошибке парсинга или валидации используется `max_retries=3`.

## 3. Использованные техники курса

В проекте применены следующие техники курса:

1. Structured output через Pydantic.
2. `field_validator` для контроля качества структуры.
3. Classification для категории, приоритета и отдела.
4. Information extraction для summary, evidence quote и escalation reason.
5. RAG по policy-документам.
6. Agent tools с сохранением trace.
7. LLM-as-judge с Pydantic `response_model` и `max_retries=3`.
8. Hallucination check: ghost quotes и fake sources.
9. Eval на размеченном наборе из 20 кейсов.

Таким образом, проект использует больше четырёх техник курса и соответствует техническому минимуму финального задания.

## 6. Challenge set и анализ ошибок

Чтобы не ограничиваться только основным eval, был создан дополнительный challenge set из 12 сложных и неоднозначных кейсов. Он нужен для проверки устойчивости системы к формулировкам, которых не было в основной выборке.

Результаты challenge set:

| Метрика | Значение |
|---|---:|
| total | 12 |
| category_accuracy | 0.75 |
| department_accuracy | 0.75 |
| quote_valid_rate | 1.0 |
| pass_rate | 0.75 |
| failed_cases | 3 |
| ghost_quote_count | 0 |
| fake_source_count | 0 |

Ошибки на challenge set:

1. `challenge_005`: “Someone used my account without permission and changed my email address.”

Ожидаемая категория: `security_privacy`.  
Предсказанная категория: `other`.  
Ожидаемый отдел: `security_team`.  
Предсказанный отдел: `support_l1`.

Причина ошибки: в keyword-based правилах не было достаточно явных признаков для фразы “without permission” и смены email как security/privacy кейса. Система не распознала это как несанкционированный доступ.

2. `challenge_006`: “Where can I find the export feature for invoices?”

Ожидаемая категория: `product_question`.  
Предсказанная категория: `billing_refund`.  
Ожидаемый отдел: `support_l1`.  
Предсказанный отдел: `billing_team`.

Причина ошибки: слово “invoices” попало в billing-keywords, хотя смысл тикета был вопросом о функции экспорта. Система отдала приоритет финансовому ключевому слову, а не вопросительной структуре.

3. `challenge_008`: “My subscription was cancelled and the latest invoice still charged my card.”

Ожидаемая категория: `subscription`.  
Предсказанная категория: `billing_refund`.  
Ожидаемый отдел: `support_l2`.  
Предсказанный отдел: `billing_team`.

Причина ошибки: тикет содержит две проблемы одновременно: подписка и списание. Текущая логика классификации выбирает одну категорию по количеству keyword hits. В данном случае billing-сигналы оказались сильнее, хотя ожидаемая маршрутизация была через subscription.

Эти ошибки показывают главное ограничение текущей версии: keyword-based классификация хорошо работает на типовых кейсах, но хуже справляется с неоднозначными обращениями, где есть несколько проблем или непрямые формулировки.

## 7. Проверка галлюцинаций

В проекте реализованы две проверки галлюцинаций.

Первая проверка - ghost quote check. Она проверяет, что evidence quote, указанная в результате, действительно содержится в исходном `issue_description`.

Вторая проверка - fake source check. Она проверяет, что все policy-документы из `used_sources` существуют в папке `input/policy_docs/`.

Результаты:

- в основном pipeline: `ghost_quote_count = 0`, `fake_source_count = 0`;
- в основном eval: `ghost_quote_count = 0`, `fake_source_count = 0`;
- в challenge set: `ghost_quote_count = 0`, `fake_source_count = 0`.

Это означает, что система не выдумывала цитаты и не ссылалась на несуществующие документы.

## 8. Выводы

Финальный проект реализует воспроизводимый AI-конвейер для классификации и маршрутизации тикетов поддержки. Система обрабатывает обращения, формирует structured output, использует RAG по базе policy-документов, вызывает agent tools, сохраняет agent trace, проверяет hallucination risks и оценивается через rule-based eval и LLM-as-judge.

Основной eval показал высокое качество на 20 размеченных кейсах: все ключевые метрики равны 1.0. Это подтверждает, что pipeline корректно работает на типовых тикетах из подготовленной выборки. При этом дополнительный challenge set показал более реалистичную устойчивость: pass rate составил 0.75, а три ошибки были связаны с неоднозначностью формулировок и пересечением категорий.

Главное практическое преимущество системы - прозрачность. Для каждого тикета сохраняется не только итоговая категория, но и путь принятия решения: какие инструменты были вызваны, какие policy-документы найдены, какая цитата использована и была ли она подтверждена.

Возможные улучшения:

1. Заменить keyword-based классификацию на LLM-based или embedding-based classifier.
2. Разрешить multi-label классификацию для тикетов с несколькими проблемами.
3. Расширить policy-документы и добавить больше примеров edge cases.
4. Использовать semantic retrieval вместо TF-IDF.
5. Сравнить несколько LLM-судей и добавить более точное ценообразование с учётом cache hit/cache miss.
