# Мониторинг дрейфа данных

Мониторинг дрейфа является одной из центральных частей MLOps-проекта. После обучения модель работает на данных, которые могут отличаться от исходного обучающего набора. Если различия становятся существенными, качество предсказаний может ухудшиться. Поэтому в проекте `rental_price_mlops` предусмотрены механизмы наблюдения за изменениями данных и поведения модели.

Первый тип мониторинга связан с data drift. Data drift означает, что распределение входных признаков изменилось. Например, в данных по аренде жилья может измениться доля объектов из определенных районов, среднее количество отзывов, типы комнат, доступность объектов или географическое распределение объявлений. Если модель обучалась на одном распределении, а новые данные приходят из другого, прогнозы могут стать менее надежными.

Для data drift нужен reference dataset. В проекте он сохраняется как `data/reference/reference.parquet`. Это базовый набор, с которым сравниваются новые входные данные. Если текущие данные сильно отличаются от reference-набора, мониторинг может поставить флаг. Поэтому подготовка reference dataset на этапе preprocessing напрямую связана с последующим мониторингом.

Второй тип мониторинга связан с concept drift. Concept drift означает, что меняется зависимость между признаками и целевой переменной. Например, раньше район или тип комнаты могли объяснять цену одним образом, а после изменений на рынке аренды эта связь стала другой. Даже если распределение признаков похоже, сама зависимость может измениться. Для фиксации concept drift обычно нужны фактические значения цены, которые появляются после предсказания.

Третий тип мониторинга связан с target drift. Target drift означает изменение распределения целевой переменной. В задаче аренды это может проявляться как общий рост или снижение цен, появление большего количества дорогих объектов или изменение сезонной структуры. Target drift особенно важен, если модель используется долго и рынок меняется.

Мониторинг в проекте связан с пользовательским интерфейсом. Страница `/monitoring-ui` нужна для показа состояния данных и флагов аномалий. В ней можно отобразить результаты сравнения текущих данных с reference-набором. Это удобно для защиты, потому что преподаватель видит не просто текстовый лог, а отдельную часть системы, отвечающую за качество данных после запуска.

Мониторинг также связан с Prometheus и Grafana. Prometheus может собирать числовые показатели, связанные с состоянием сервиса и результатами проверок. Grafana может визуализировать эти показатели. Например, можно показать, сколько запросов было обработано, сколько записей вызвали подозрение, какие признаки сильнее всего отличаются от reference dataset. Даже если в учебной версии часть метрик упрощена, архитектурная идея остается правильной.

Особое значение имеют флаги аномалий. Они нужны для быстрого понимания состояния системы. Если все флаги нормальные, модель можно продолжать использовать. Если появились признаки drift, нужно разобраться в причинах. Возможно, изменился источник данных, появились новые типы объектов или пользователь начал отправлять некорректные значения. В зависимости от причины может потребоваться исправление данных, обновление preprocessing или переобучение модели.

Мониторинг не решает проблему сам по себе. Он только сообщает, что система работает в новых условиях. Следующий шаг связан с retrain. Если дрейф устойчивый и влияет на качество, модель нужно переобучить на более актуальных данных. Поэтому мониторинг и переобучение образуют единую цепочку: сначала система обнаруживает изменение, потом команда принимает решение, затем запускается обновление модели.

В отчете по проекту мониторинг можно описывать как компонент, который снижает риск незаметного ухудшения качества. Без мониторинга модель может продолжать выдавать предсказания, хотя данные уже сильно отличаются от обучающих. С мониторингом команда получает сигнал и может реагировать. Это особенно важно для MLOps, где жизненный цикл модели продолжается после первого запуска.

Для домашнего задания по RAG этот документ полезен для вопросов о data drift, concept drift, target drift, reference dataset, monitoring UI и флагах аномалий. Он также хорошо подходит для multi-hop вопросов вместе с документами про переобучение, API и Docker-инфраструктуру.
