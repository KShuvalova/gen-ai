# Docker Compose, Prometheus и Grafana

Docker Compose в проекте `rental_price_mlops` используется для запуска нескольких сервисов в единой инфраструктуре. В обычном ML-проекте часто достаточно запустить один Python-скрипт. В MLOps-проекте этого мало, потому что система включает API, мониторинг, визуализацию метрик и иногда дополнительные сервисы. Docker Compose помогает описать эти компоненты и запускать их вместе.

Один из ключевых сервисов в проекте связан с API. Он отвечает за инференс модели и пользовательские страницы. Другие сервисы связаны с мониторингом. Prometheus используется для сбора метрик, а Grafana используется для их визуализации. В проекте Prometheus доступен на порту 9090, а Grafana доступна на порту 3000. Это стандартная связка для наблюдаемости сервисов.

Prometheus нужен для регулярного сбора числовых показателей. Он может получать информацию о состоянии API, количестве запросов, времени ответа, ошибках, флагах мониторинга и других метриках. В MLOps-контексте метрики могут быть связаны не только с техническим состоянием сервиса, но и с данными. Например, можно фиксировать показатели дрейфа, количество подозрительных записей или изменения в распределении признаков.

Grafana нужна для удобной визуализации. Даже если все метрики доступны в текстовом виде, пользователю или преподавателю легче воспринимать их через графики и панели. На дашборде можно показать динамику запросов, состояние сервиса, признаки аномалий и результаты мониторинга. Поэтому Grafana делает проект более демонстрационным и ближе к промышленной практике.

Docker Compose также помогает решить проблему воспроизводимости окружения. Если все сервисы описаны в `docker-compose.yml`, то их можно запустить одной командой, не настраивая каждый компонент вручную. Это особенно важно для учебного проекта, который нужно показать на другой машине или в Codespaces. Чем меньше ручных действий требуется для запуска, тем выше качество MLOps-решения.

В проекте сборка Docker Compose занимала заметное время, около нескольких минут. Это нормально, потому что контейнеры могут устанавливать зависимости и собирать окружение. В одном из состояний проекта команда `docker compose ps` показывала пустой список сервисов. Это означало, что в тот момент сервисы не были запущены или завершились. Такая проверка полезна, потому что она позволяет быстро понять, работает ли инфраструктура.

Также были проблемы с Docker daemon. Если Docker не запущен или работает нестабильно, compose-команды не смогут создать и запустить контейнеры. В таком случае нужно сначала убедиться, что Docker Desktop работает, а затем повторить запуск. Для Mac это особенно важно, потому что Docker зависит от фонового приложения и виртуализированного окружения.

Связка Docker, Prometheus и Grafana важна для защиты проекта. Она показывает, что модель не просто обучена, а помещена в инфраструктуру. Можно открыть API, затем открыть Prometheus, затем показать Grafana. Это демонстрирует переход от ML к MLOps: система не только делает предсказания, но и контролируется после запуска.

Docker также связан с хранением артефактов. Если модель, данные или логи должны сохраняться между запусками контейнеров, нужно правильно настроить volumes. Иначе после перезапуска контейнера часть результатов может исчезнуть. Поэтому инфраструктурная часть должна быть согласована с DVC, отчетами и папками проекта.

Для RAG-корпуса этот документ является источником информации о контейнерах, мониторинге и визуализации. Вопросы про Prometheus, Grafana, порт 9090, порт 3000, Docker daemon и `docker compose ps` должны находить именно этот документ. В multi-hop вопросах он может использоваться вместе с документом про DVC или мониторинг, потому что инфраструктура связана с хранением артефактов и контролем дрейфа.
